### BUG-001: Server does not validate history array structure

**Severity**: LOW
**Source**: Code review Pass 1 (structural)
**File**: `server/src/routes/chat.ts:36`

**Description**: The `/api/chat` route validates the `message` field but passes the `history` array directly to `prompt-builder` without checking each item's structure (role, content fields). Malformed history items propagate to the DeepSeek API, causing a cryptic error.

**Spec basis**: REQ-007 — server must validate history array structure

**Steps to reproduce**:
1. Send POST /api/chat with `{ "message": "hi", "history": [{ "role": "user" }] }` (missing content)
2. Server accepts the request
3. prompt-builder passes the incomplete object to DeepSeek API
4. DeepSeek returns error, user sees "Failed to get AI response"

**Regression test**: `quality/test_regression.ts` — `BUG-001: should reject history items missing content field`

**Fix**: Add validation loop in `routes/chat.ts` after message validation:
```typescript
if (history) {
  if (!Array.isArray(history)) return res.status(400).json(...);
  for (const item of history) {
    if (!item.role || !item.content) return res.status(400).json(...);
  }
}
```

### BUG-002: SSE error events silently swallowed

**Severity**: HIGH
**Source**: Code review Pass 2 (requirements)
**File**: `client/src/lib/api.ts:175-178`

**Description**: The SSE parser's inner catch block catches ALL errors — including `throw new Error(data.error)` from server error events. The error is silently discarded. The user never sees error messages sent by the server.

**Spec basis**: REQ-001 — SSE error events must be visible to user

**Steps to reproduce**:
1. Server sends `data: {"error":"Rate limit exceeded"}`
2. Client parses line, hits `if (data.error) throw new Error(data.error)`
3. The throw is caught by the inner catch block `catch { // skip malformed lines }`
4. Error never reaches `callbacks.onError()`
5. User sees nothing, loading continues indefinitely

**Regression test**: `quality/test_regression.ts` — `BUG-002: should handle error events from server`

**Fix**: Separate error handling from parse error handling:
```typescript
for (const line of lines) {
  if (line.startsWith("data: ")) {
    let data;
    try {
      data = JSON.parse(line.slice(6));
    } catch {
      continue; // skip malformed JSON
    }
    if (data.token) callbacks.onToken(data.token);
    if (data.error) { callbacks.onError(new Error(data.error)); return; }
    if (data.usage) finish();
  }
}
```

### BUG-003: SSE fetch has no timeout

**Severity**: HIGH
**Source**: Code review Pass 1 (structural)
**File**: `client/src/lib/api.ts:137-194`

**Description**: The `fetch()` call for SSE streaming uses no AbortController or timeout. If the server hangs (DeepSeek stall, Vercel cold start > 30s), the client connection stays open indefinitely. The user sees a stuck loading state with no feedback.

**Spec basis**: REQ-002 — SSE fetch must have a timeout

**Steps to reproduce**:
1. Send chat message
2. Server starts processing but DeepSeek API never responds
3. Client's `reader.read()` never resolves
4. User sees loading state forever

**Regression test**: `quality/test_regression.ts` — `BUG-003: should use AbortController with timeout`

**Fix**: Add AbortController with 30s timeout:
```typescript
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 30000);
try {
  const res = await fetch(`${API_BASE}/chat`, {
    ...options,
    signal: controller.signal,
  });
} finally {
  clearTimeout(timeout);
}
```

### BUG-004: No retry for transient DeepSeek failures

**Severity**: MEDIUM
**Source**: Code review Pass 1 (structural)
**File**: `server/src/lib/chat/deepseek-client.ts:15-40`

**Description**: `streamFromDeepSeek` makes a single API call with no retry logic. Network errors, 429 rate limits, and 5xx server errors from DeepSeek propagate immediately as failures. No exponential backoff.

**Spec basis**: REQ-006 — transient failures must retry

**Steps to reproduce**:
1. DeepSeek API returns 429 (rate limited)
2. Error propagates to route handler
3. User sees "Failed to get AI response"

**Regression test**: `quality/test_regression.ts` — `BUG-004: should retry on network error`

**Fix**: Add retry wrapper with exponential backoff:
```typescript
async function withRetry<T>(fn: () => Promise<T>, retries = 2): Promise<T> {
  for (let i = 0; i <= retries; i++) {
    try { return await fn(); }
    catch (err) {
      if (i === retries) throw err;
      await new Promise(r => setTimeout(r, (i + 1) * 1000));
    }
  }
  throw new Error("unreachable");
}
```

### BUG-005: Ghost empty assistant message on send failure

**Severity**: MEDIUM
**Source**: Code review Pass 2 (requirements)
**File**: `client/src/hooks/useChat.ts:34-40`

**Description**: An empty assistant message is added to the store BEFORE the API call. If `chatApi.sendStream` throws immediately (network error, 400 response), the empty assistant message remains in the chat history permanently. The user sees a blank assistant bubble.

**Spec basis**: REQ-003 — failed sends must not leave ghost messages

**Steps to reproduce**:
1. User types a message and sends
2. Network is offline
3. Empty assistant bubble appears and stays

**Regression test**: `quality/test_regression.ts` — `BUG-005: should remove empty message on error`

**Fix**: Remove the empty message on error:
```typescript
try {
  // ... API call
} catch (err: any) {
  // Remove the empty assistant message
  useChatStore.getState().messages = useChatStore.getState().messages.filter(
    (m) => m.id !== assistantId
  );
  setError(err.message || "Failed to send message");
}
```

### BUG-006: Missing API key not validated at startup

**Severity**: MEDIUM
**Source**: Code review Pass 1 (structural)
**File**: `server/src/lib/chat/deepseek-client.ts:3-6`

**Description**: `new OpenAI({ apiKey: process.env.DEEPSEEK_API_KEY || "" })` silently initializes with an empty string if the env var is not set. Every API call will fail with 401, but the error surfaces as a generic "Failed to get AI response" with no indication that the root cause is a missing API key.

**Spec basis**: REQ-005 — missing key must fail at startup

**Steps to reproduce**:
1. Deploy without setting DEEPSEEK_API_KEY
2. Any chat request fails
3. Log shows generic "Chat error" with no actionable message

**Regression test**: `quality/test_regression.ts` — `BUG-006: should throw when DEEPSEEK_API_KEY is empty`

**Fix**: Validate at module initialization:
```typescript
if (!process.env.DEEPSEEK_API_KEY) {
  throw new Error("DEEPSEEK_API_KEY environment variable is required");
}
```
