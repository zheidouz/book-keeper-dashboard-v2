# Spec Audit Protocol: AI Bookkeeper Chatbot — Council of Three

> Generated by Quality Playbook v1.5.6 — Andrew Stellman
> Date: 2026-06-01 · Project: book-keeper-dashboard-v2

---

## Audit Prompt

Audit the AI Bookkeeper Chatbot implementation against the PRD (`docs/prd-ai-chatbot.md`) and the quality requirements (`quality/EXPLORATION.md`).

**Files to audit**:
- `server/src/routes/chat.ts` — POST /api/chat handler
- `server/src/lib/chat/deepseek-client.ts` — DeepSeek streaming
- `server/src/lib/chat/prompt-builder.ts` — system prompt
- `server/src/lib/chat/form-retriever.ts` — DB query
- `server/src/lib/chat/sse-responder.ts` — SSE write
- `client/src/lib/api.ts` — SSE parser
- `client/src/hooks/useChat.ts` — message send hook
- `client/src/stores/chat-store.ts` — state store

**Scrutiny areas**:
1. **Input validation** — Does every trust boundary validate inputs? (HTTP body, SSE data, user message)
2. **Error handling** — Are all error paths visible to the user? Are errors silently swallowed?
3. **Resource lifecycle** — Are streams, connections, and timeouts properly managed?
4. **State consistency** — Does the Zustand store ever enter an inconsistent state (ghost messages, stuck loading)?
5. **Security** — Is the rate limiter correctly scoped? Is the API key validated? Any XSS vectors?
6. **Spec compliance** — Does the implementation match the PRD's requirements for streaming, error handling, and rate limiting?

**Guardrails**:
- Line numbers are mandatory for every finding
- Read function bodies, not just signatures
- If unsure: flag as QUESTION, not BUG

## Triage Process

1. Collect all findings from 3 auditors
2. Merge by confidence level:
   - **3/3 agreement**: Auto-confirm
   - **2/3 agreement**: Confirm
   - **1/3 (minority)**: Re-investigate with verification probe
3. For each minority finding, read the specific code location and make explicit CONFIRMED/FALSE-POSITIVE determination
4. Record in `quality/spec_audits/YYYY-MM-DD-triage.md`

## Verification Probes

Every verification probe must produce executable evidence:
- **For rejections**: Write assertion that PASSES, proving the finding is wrong
- **For confirmations**: Write assertion that FAILS, proving the bug exists
- Every assertion must cite exact line number

## Output

- Individual auditor reports: `quality/spec_audits/2026-06-01-auditor-1.md` (etc.)
- Triage synthesis: `quality/spec_audits/2026-06-01-triage.md`
