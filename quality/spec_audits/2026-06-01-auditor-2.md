# Auditor 2 Report — State Management & UX Focus

> Date: 2026-06-01 · Focus: UI state, streaming UX, error visibility

## Findings

### FINDING-2A: No retry mechanism for failed messages (CONFIRMED)
**File**: `client/src/hooks/useChat.ts:34-40`
When an API call fails, the empty assistant message stays with no retry button. The user must start a new chat. This is a UX gap — the implementation plan mentions retry in the architecture but it was never implemented.

### FINDING-2B: Loading state stuck on connection drop (CONFIRMED)
**File**: `client/src/lib/api.ts:137-194`
If the ReadableStream reader throws mid-read (connection drop), `callbacks.onDone()` is never called, so `setLoading(false)` never fires. The loading indicator spins forever. This is a blocking UX bug.

### FINDING-2C: Rate limiter is per-instance, not global (CONCERN)
**File**: `server/src/routes/chat.ts:9-13`
`express-rate-limit` is in-memory. On Vercel serverless, each cold start creates a new rate limiter instance. A user who exhausts their limit on one instance can make new requests on a different instance. The rate limit is advisory, not enforced. This is a known limitation of serverless rate limiting.

### FINDING-2D: DeepSeek model name hardcoded (CONCERN)
**File**: `server/src/lib/chat/deepseek-client.ts:15`
`model: "deepseek-chat"` is hardcoded. If DeepSeek changes their model identifiers, the chat breaks silently. Should be configurable via env var.

### FINDING-2E: Empty state has no error recovery path (CONFIRMED)
**File**: `client/src/components/chat/ChatPanel.tsx:25-35`
The empty state shows "Ask me about BIR forms" with suggestions. If the first message fails, the error appears briefly but the empty assistant message persists. No retry prompt is shown.
