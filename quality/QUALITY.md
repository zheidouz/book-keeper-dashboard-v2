# Quality Constitution: AI Bookkeeper Chatbot

> Generated by Quality Playbook v1.5.6 — Andrew Stellman
> Date: 2026-06-01 · Project: book-keeper-dashboard-v2

---

## 1. Purpose

Quality for the AI Bookkeeper Chatbot means: **users receive accurate, streamed answers to BIR form questions without data loss, ghost states, or silent failures.** Fitness for use means the chatbot is available when needed, errors are visible and actionable, and no user action (typing a question) produces a permanently broken UI state.

## 2. Coverage Targets

| Module | Target | Rationale |
|--------|--------|-----------|
| SSE streaming (client) | 100% of error paths | Silent failures are the worst UX outcome — user sees nothing and doesn't know why |
| SSE streaming (server) | 100% of write paths | `res.write()` failures must not silently continue |
| Message lifecycle (send → stream → done) | 100% of state transitions | Ghost messages, stuck loaders, and double-sends are UI integrity bugs |
| Rate limiting | 100% of auth states | Unauthenticated users must not share rate limit buckets |
| Prompt builder | 90%+ of input combinations | Malformed history, empty messages, injection attempts |
| DeepSeek client | 100% of failure modes | Network errors, auth errors, streaming errors, timeouts |

## 3. Coverage Theater Prevention

- **No "test that imports work" tests** — every test must send a message, parse a response, or assert a behavioral outcome
- **No mock-that-returns-configured-value tests** — SSE stream tests must use real ReadableStream instances, not pre-baked callbacks
- **No "the function didn't throw" assertions** — every assertion must check return values, state changes, or callback invocations
- **No assertion-free streaming tests** — a test that calls `sendStream()` without asserting that `onToken` or `onDone` fired is a coverage theater test

## 4. Fitness-to-Purpose Scenarios

### Scenario 1: Server sends error mid-stream
**What happened**: Server's DeepSeek client throws mid-response after SSE headers are sent. The server writes `data: {"error":"..."}` and closes. The client's SSE parser receives the error event.
**What should happen**: `onError` fires with the error message. Loading stops. Error displays in chat panel.
**How to verify**: Mock a partial SSE stream with an error event injected mid-stream. Assert `onError` fires.

### Scenario 2: Client network drops mid-stream
**What happened**: User receives partial response, then WiFi drops. The client's `reader.read()` throws a network error.
**What should happen**: `onError` fires. Loading stops. The partial message remains with a visual error indicator. User can retry.
**How to verify**: Create a ReadableStream that aborts mid-read. Assert `onError` fires with network error.

### Scenario 3: DeepSeek API returns 401
**What happened**: DEEPSEEK_API_KEY is expired or invalid. The server calls DeepSeek API, gets 401 Unauthorized.
**What should happen**: Server catches the error. If SSE headers not yet sent, returns 500 JSON. If headers sent, writes SSE error event. User sees error message.
**How to verify**: Mock OpenAI SDK to throw 401. Assert catch block fires and appropriate error is returned.

### Scenario 4: Rate limit exceeded
**What happened**: User sends 31 requests in 60 seconds. The 31st request is blocked.
**What should happen**: Server returns 429 with `{ success: false, error: "Rate limit exceeded..." }`. Client shows error in chat panel. No API call to DeepSeek.
**How to verify**: Send 31 rapid requests. Assert 31st returns 429.

### Scenario 5: User sends message while already streaming
**What happened**: User clicks Send rapidly twice without waiting for the first response.
**What should happen**: InputBar is disabled while `isLoading` is true. The second click is ignored.
**How to verify**: Check `disabled` prop on InputBar when `isLoading=true`. Verify `onSend` is not called.

### Scenario 6: Unauthenticated user accesses chat
**What happened**: No valid JWT. `req.authUser` is undefined.
**What should happen**: Rate limiter keys to "anonymous". Chat endpoint should reject with 401.
**How to verify**: Send request without Authorization header. Assert 401 response.

### Scenario 7: Missing DEEPSEEK_API_KEY on startup
**What happened**: Server starts without DEEPSEEK_API_KEY environment variable.
**What should happen**: Module initialization throws. Server fails to start or logs a critical error before first request.
**How to verify**: Clear DEEPSEEK_API_KEY env var. Import module. Assert throws.

### Scenario 8: History contains malformed items
**What happened**: Client sends `history: [{ role: "user" }]` (missing content field) or `history: [{ content: "hi" }]` (missing role field).
**What should happen**: Server validates each history item. Invalid items are dropped or return 400.
**How to verify**: Send malformed history. Assert server returns 400 or drops invalid items.

## 5. AI Session Quality Discipline

1. Every AI session working on the chat module must read `quality/QUALITY.md` and `quality/REQUIREMENTS.md` before making changes.
2. Any change to SSE streaming code must include a test for the error path — not just the happy path.
3. Any change to `api.ts` SSE parser must not widen the catch block beyond known error types.
4. New env vars must be validated at startup, not lazily at request time.

## 6. The Human Gate

- **Rate limit thresholds**: The default 30 req/min is a starting point. Tune based on actual usage patterns.
- **DeepSeek model selection**: `"deepseek-chat"` is a placeholder model name. Verify the correct identifier with DeepSeek's current API docs before production.
- **Retry strategy**: 2 retries with 1s/2s backoff is a reasonable starting point. Adjust based on observed DeepSeek API latency characteristics.
