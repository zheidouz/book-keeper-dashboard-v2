## Domain and Stack

**Project**: AI Bookkeeper Chatbot — a floating chat assistant embedded in a Philippine accounting dashboard. Users ask questions about BIR forms (tax form codes, deadlines, filing frequencies) and receive streaming AI responses.

**Language**: TypeScript (both client & server)
**Frontend**: React 18 + Vite + Tailwind CSS + Zustand + Lucide icons
**Backend**: Express 5 + Drizzle ORM + PostgreSQL (Neon)
**Auth**: Clerk JWT verification
**LLM**: DeepSeek V4 Flash via OpenAI-compatible SDK
**Deploy**: Vercel serverless (api/index.ts)

## Open Exploration Findings

### F-01: Server does not validate message body structure beyond `message` field
**File**: `server/src/routes/chat.ts:25-34`
**Finding**: The route only checks `message` exists, is a string, and is non-empty. It does NOT validate the `history` array structure — history items are passed directly to the prompt builder without checking that each item has `role` and `content` fields. Malformed history (e.g., `[{role: "user"}]` without content, or `[{content: "hi"}]` without role) would propagate to the LLM API, potentially causing a 400 error from DeepSeek with an unhelpful error message reaching the client.
**Cross-module**: Chat route → prompt-builder → deepseek-client. The validation gap at the route level affects two downstream modules.

### F-02: Rate limiter key falls back to "anonymous" when authUser is missing
**File**: `server/src/routes/chat.ts:13`
**Finding**: `keyGenerator` uses `authUser?.id || "anonymous"`. All authenticated users have individual rate limit buckets. But unauthenticated requests (no valid JWT) collapse into a single shared "anonymous" bucket, meaning one abusive unauthenticated source blocks all others. Additionally, if `clerkAuth` middleware fails silently (e.g., expired token that doesn't throw), `authUser` could be undefined, routing the user into the shared anonymous bucket with no feedback.
**Cross-module**: The rate limiter at `route level` depends on `clerkAuth` middleware from `api/index.ts:53` having run first — this ordering is correct but fragile.

### F-03: SSE write errors are silently swallowed
**File**: `client/src/lib/api.ts:175-178`
**Finding**: The inner try/catch around `JSON.parse(line.slice(6))` has an empty catch block that catches ALL errors — including legitimate JSON parse failures AND `throw new Error(data.error)`. When `data.error` is thrown inside the try block, it's caught by the same catch and silently skipped. The error is never propagated to the caller. This means server-side error events sent mid-stream are lost without the user knowing.
**Cross-module**: Server writes `writeSSEError(res, err.message)` at sse-responder.ts:7, but the client's api.ts catch block silently discards it.

### F-04: No timeout on SSE stream fetch
**File**: `client/src/lib/api.ts:127-194`
**Finding**: `fetch()` is called with no abort signal or timeout. If the server hangs mid-stream (e.g., DeepSeek API stalls, Vercel cold start exceeds 30s), the client connection stays open indefinitely. The `reader.read()` loop will never resolve. The user sees an empty assistant bubble with no error feedback.
**Cross-module**: deepseek-client.ts has no timeout on the OpenAI SDK call either. Two layers of hanging.

### F-05: Empty response from DeepSeek creates ghost assistant message
**File**: `client/src/hooks/useChat.ts:34-40`
**Finding**: An empty assistant message is added to the store BEFORE the stream starts. If `chatApi.sendStream` throws immediately (network error, 400 response), the empty assistant message remains in the chat history permanently. The user sees a blank assistant bubble that never populated.

### F-06: Message input cleared but history captured before user message is added
**File**: `client/src/hooks/useChat.ts:25-29`
**Finding**: The `send()` function calls `useChatStore.getState().messages` to build history AFTER adding the user message to the store (line 21: `addMessage(userMsg)`). Since Zustand updates are synchronous, the history correctly includes the user's own message. **Confirmed correct** — the stale closure fix is working.

### F-07: Markdown list rendering regex has edge case
**File**: `client/src/components/chat/MessageBubble.tsx:21-25`
**Finding**: The list regex `/(<li>[\s\S]*?<\/li>)(\s*<li>)/g` and `/(<li>[\s\S]*?<\/li>\s*)+/g` work for consecutive items but don't handle nested lists, ordered lists, or lists with blank lines between items. A response with mixed content (text, blank line, then a list) will render the list items as `<li>` but may not wrap them in `<ul>`. This is a rendering quality issue, not a security bug.

### F-08: No role-based access control for the chat endpoint
**File**: `server/src/routes/chat.ts:14-22`
**Finding**: The `/api/chat` route has no `requireRole()` middleware. Any authenticated user (including `encoder` role) can call the API, incurring token costs. The PRD specifies the chatbot is for "All roles" — so this is not a bug per the PRD. However, there's no cost-control mechanism per role; an encoder could abuse the endpoint to rack up token spend.

## Quality Risks

### R-01 (HIGH): DeepSeek API key exposure risk on error
**File**: `server/src/lib/chat/deepseek-client.ts:3-6`
**Risk**: The OpenAI client is initialized with `apiKey: process.env.DEEPSEEK_API_KEY || ""`. If the env var is unset, the client sends requests with an empty API key. DeepSeek's API returns a 401/403 with a stack trace that may leak configuration details. Worse, the client silently initializes even with a missing key — no startup-time validation warns the developer.

### R-02 (HIGH): Express 5 `res.write()` after `res.end()` could crash
**File**: `server/src/routes/chat.ts:47-51`
**Risk**: If `streamFromDeepSeek` calls `onDone` callback which calls `res.end()`, but a race condition causes additional SSE events to arrive from DeepSeek after the stream's `done` event, `writeSSE` would call `res.write()` after `res.end()`. In Express 5 (which is used), writing to a closed response throws an error. The catch block at line 53 would handle it, but the error message "Cannot write after end" would be confusing in logs.

### R-03 (MEDIUM): DeepSeek model string might not match deployment
**File**: `server/src/lib/chat/deepseek-client.ts:15`
**Risk**: The model string `"deepseek-chat"` is hardcoded. DeepSeek has used different model identifiers for V4 Flash (`"deepseek-chat"`, `"deepseek-v4-flash"`, `"deepseek-ai/DeepSeek-V4-Flash"`). If the API changes its model identifier, the chat breaks silently — the API returns a 404/model-not-found error that surfaces as "Failed to get AI response" to the user with no actionable diagnostic.

### R-04 (MEDIUM): No rate limit on the rate limiter key generator
**File**: `server/src/routes/chat.ts:13`
**Risk**: `express-rate-limit` with `keyGenerator` that returns `"anonymous"` for unauthenticated requests means ALL anonymous requests share one bucket. A single attacker or misconfigured client can exhaust the anonymous bucket, blocking all unauthenticated users. Additionally, the rate limiter is in-memory — on Vercel serverless, each cold start creates a new rate limiter instance, resetting all counters.

### R-05 (LOW): `parseInt` without radix on env vars
**File**: `server/src/routes/chat.ts:12`, `server/src/lib/chat/deepseek-client.ts:7`
**Risk**: `parseInt(process.env.CHAT_RATE_LIMIT || "30")` and `parseInt(process.env.CHAT_MAX_TOKENS || "500")` don't pass a radix. If someone sets `CHAT_RATE_LIMIT=08`, `parseInt` in some environments interprets leading zero as octal (legacy behavior, Edge cases). Modern Node.js treats it as decimal, but it's a hygiene issue.

## Pattern Deep Dive — Fallback and Degradation Path Parity

### Layer 1: DeepSeek API call failure
**File**: `server/src/lib/chat/deepseek-client.ts:15-40`
**Behavior**: If `client.chat.completions.create()` throws (network error, timeout, auth error), the exception propagates to routes/chat.ts catch block. Response: 500 JSON or SSE error event. No retry logic.

**Layer 2: Partial stream failure mid-response**
**File**: `server/src/lib/chat/deepseek-client.ts:24-36`
**Behavior**: The `for await (const chunk of stream)` loop iterates over SSE chunks from DeepSeek. If the connection drops mid-stream, the `for await` loop throws, caught by routes/chat.ts catch block. By this point, SSE headers are sent and some tokens may have been delivered. The catch writes an SSE error event, but the user has a partially-filled assistant message with an error appended.

**Layer 3: Client-side fetch abort**
**File**: `client/src/lib/api.ts:137-180`
**Behavior**: No abort signal. No timeout. No connection retry. A dropped connection mid-stream leaves the user with a partial assistant message and the loading indicator stuck (because `finish()` is never called).

**Layer 4: Empty response handling**
**File**: `client/src/hooks/useChat.ts:34-40`
**Behavior**: Empty assistant message created before API call. If the API throws immediately, the empty message persists. No mechanism to remove or retry a failed message.

**Verdict**: Degradation is not graceful at any layer. Three compounding gaps: (1) no retry, (2) no client timeout, (3) partial messages persist on failure with no retry mechanism. Each layer degrades independently and the user has no recovery path.

## Pattern Deep Dive — Dispatcher Return-Value Correctness

### `res.write()` return value ignored
**File**: `server/src/lib/chat/sse-responder.ts:4`
**Finding**: `res.write()` returns a boolean — `true` if the data was buffered internally, `false` if the data could not be written (stream is full, connection lost). The return value is not checked. If the client disconnects, `res.write()` may return `false` or throw. The code continues writing SSE events to a defunct connection.

**Client-side SSE parser error swallowing**
**File**: `client/src/lib/api.ts:175-178`
**Finding**: The inner catch block `catch { // skip malformed lines }` is too broad. It catches:
1. Legitimate JSON parse errors from malformed SSE data
2. The `throw new Error(data.error)` from server error events
3. Any other unexpected errors in the parsing loop

When `data.error` is thrown (line 173), the catch block silently discards it. The user never sees the error.

**Cross-layer trace**: Server error → `writeSSEError(res, err)` → client receives `data: {"error":"..."}` → JSON.parse succeeds → `if (data.error) throw new Error(data.error)` → caught by `catch {}` → silently dropped. The error path is designed (server writes error events) but the client's catch block nullifies it.

## Pattern Deep Dive — API Surface Consistency

### Request validation asymmetry
**Server side** (`routes/chat.ts:25-30`): Validates `message` field (required, string, non-empty). Does NOT validate `history` array.
**Client side** (`api.ts` send body): Always sends `{ message, history }` where history is an array of `{ role, content }`.
**Gap**: The server trusts the history array implicitly. Malformed history items propagate to the prompt builder, which passes them to DeepSeek. The LLM API will reject malformed messages, but the error surfaces as a generic 500.

### Error format inconsistency
**Server side** (`routes/chat.ts:27`): Validation error returns `{ success: false, error: "Message is required" }` — JSON body.
**Server side** (`routes/chat.ts:52-57`): Runtime error after SSE headers sent returns SSE event `data: {"error":"..."}`.
**Client side** (`api.ts:148-151`): Non-200 responses parsed as `{ success, error }` JSON.
**Client side** (`api.ts:175-178`): SSE error events silently swallowed.
**Gap**: Two different error formats (JSON vs SSE event) depending on whether headers were sent. The client handles one but not the other.

### Prompt injection surface
**File**: `server/src/lib/chat/prompt-builder.ts:23-52`
**Finding**: The system prompt includes `User role: ${input.userRole || "unknown"}` — this is server-controlled (from JWT), so not injectable. However, the `input.userMessage` at line 50 is injected directly into the messages array sent to DeepSeek. While the LLM is instructed not to obey user commands overriding the system prompt (prompt injection is a general LLM problem), there's no content filtering or sanitization of user messages. A user could attempt prompt injection to override the system behavior.

## Derived Requirements

### REQ-001: SSE error events must be visible to the user
**Tier**: 3 (code-is-the-spec)
**File**: `client/src/lib/api.ts`
**Conditions**: When the server sends a `data: {"error": "..."}` SSE event, the client must surface that error message to the user via `callbacks.onError()`. The error must not be silently swallowed.

### REQ-002: SSE fetch must have a timeout
**Tier**: 3
**File**: `client/src/lib/api.ts`
**Conditions**: The fetch request must use an AbortController with a timeout (default: 30s). The timeout must trigger `onError` with "Request timed out".

### REQ-003: Failed sends must not leave ghost messages
**Tier**: 3
**File**: `client/src/hooks/useChat.ts`
**Conditions**: If the API call throws before any tokens are received, the empty assistant message must be removed from the chat store. The user must see the error message instead.

### REQ-004: Chat endpoint must reject unauthenticated requests
**Tier**: 3
**File**: `server/src/routes/chat.ts`
**Conditions**: If `req.authUser` is null/undefined, return 401. Do not fall back to anonymous rate limiting.

### REQ-005: Missing DEEPSEEK_API_KEY must fail at startup
**Tier**: 3
**File**: `server/src/lib/chat/deepseek-client.ts`
**Conditions**: If `DEEPSEEK_API_KEY` env var is unset or empty, throw at module load time — do not silently initialize with empty key.

### REQ-006: Transient DeepSeek failures must retry
**Tier**: 4 (informal docs)
**File**: `server/src/lib/chat/deepseek-client.ts`
**Conditions**: Network errors and 5xx responses from DeepSeek API must trigger up to 2 retries with exponential backoff (1s, 2s).

### REQ-007: Server must validate history array structure
**Tier**: 3
**File**: `server/src/routes/chat.ts`
**Conditions**: If `history` is provided, each item must have `role` (string, one of "user"/"assistant") and `content` (string, non-empty). Invalid items must be silently dropped or return 400.

### REQ-008: Client must handle connection drop mid-stream
**Tier**: 3
**File**: `client/src/lib/api.ts`
**Conditions**: If the `reader.read()` throws (connection dropped), `callbacks.onError` must be called. The loading indicator must stop.
